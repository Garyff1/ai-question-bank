## 2. 已完成的模拟闭环

`登录 → 获取报价 → 创建订单 → 模拟支付 → 创建生成任务 → Fake AI 生成 → 保存结果 → 订单完成 → 记录影子计费`

失败闭环覆盖：

`模拟支付成功 → AI 失败 → 自动重试 → 仍失败 → 模拟退款 → 订单已退款`

自动化测试覆盖重复回调、金额不一致、重复订单、重复退款、越权查询和功能开关关闭。Android API 36 模拟器也完成一次真实 UI 操作闭环，明确显示“测试环境、不会真实扣款”。

## 3. 微信支付准备度

已完成：配置模型、Provider 接口、订单/回调/退款边界、功能开关和安全禁用。

未完成：

- 微信开放平台与商户号资料
- 官方 Android SDK
- 服务端统一下单
- API v3 签名、验签、证书和回调解密
- 真机唤起微信
- 沙箱/小额真实支付与原路退款
- 重复通知、网络中断和证书轮换实测

结论：**适配层完成，Mock 测试通过，真实微信支付未验证。**

## 4. 支付宝准备度

已完成：配置模型、Provider 接口、订单/异步通知/退款边界、功能开关和安全禁用。

未完成：

结论：**适配层完成，Mock 测试通过，真实支付宝支付未验证。**

## 5. 正式支付开放顺序

1. 关闭所有商业发布阻断项。
2. 部署正式 HTTPS 后端、PostgreSQL、任务队列和共享限流。
3. 分别完成微信、支付宝沙箱或测试环境。
4. 完成小额支付、取消、超时、重复通知、生成失败和原路退款。
5. 核对订单、资金、模型任务和审计日志四方一致性。
6. 先对内部测试账号开放，再小范围灰度。
7. 最后由服务器单独开启具体通道；`REAL_CHARGE_ENABLED` 仍需人工确认。
